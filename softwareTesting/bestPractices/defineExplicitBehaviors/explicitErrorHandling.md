Explicit Error Handling

When failures occur, should I return false or an exception?
How do I make it clear exactly what failure happened to the client? ^EIfvxmpm

BAD: return used to handle unexpected events/failures ^whmDfsFC

BAD: exception used to handle business logic validation failures ^glvg32lH

Rule of thumb:
Return an error result (either false or failure Result object, see below) only for expected failures/failures we know can happen:
- input validation failures (from a business logic perspective, NOT from code wise e.g. an email not containing @, NOT like for e.g. input is null while the argument is marked not nullable)
- business logic dependencies unavailable e.g. data not existing in another table needed for this operation
- if the operation did not succeed

Else, use Exceptions i.e. events we don't expect
- db unreachable, network down
- input is wrong type code wise ^iDIah4La

VS. ^0lbS7EKm

When throwing exceptions, make it clear what happened e.g. here, Use internal library ThrowIfNull with an argument to let caller know the exception reason ^4xpcBIHS

Use Result pattern to be able to return operation results with details! ^4oECvT6D

When it makes sense, add also an object/type in your return, so tests can verify behavior easily given different situations, even if the operation is just yes/no

e.g. now can check also not just if the delivery was successful, but also that e.g. deliveryResult.Value!.Subject, has some default keywords in it, etc.

We don't have to mock e.g. deliveryService.deliver().And.Return(...) to be at least sure the message to be sent has the default values! ^QNaiAsPz

Rule of thumb:
If it makes sense, always try to return not just the operation result (success/fail) but also the output of the operation itself/any details, in a structured manner
e.g. Use the Result pattern see below; this pattern is nice because it is type agnostic, you can return any output with any type! ^pqvSa5fb
