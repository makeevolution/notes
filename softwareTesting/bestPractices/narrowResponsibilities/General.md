CreatingEffectiveInjectionPoints 

https://courses.dometrain.com/courses/take/from-zero-to-hero-writing-testable-code-in-csharp/lessons/63758905-creating-effective-injection-points

- Injection can be in method or in constructor
- Inject in constructor if the injectable is to be used in many other places
- For testability, try to inject always an interface not a concrete object, so that we can replace the injectable with a fake implementation for testing
- This is especially important if the injected object is a static class/hard-requirement outside dependency; create an interface that implements the required methods
  and inject this interface instead
- This is also known as Adapter design pattern. Example with DateTime dependency (injecting DateTime)
- THis makes it possible to create a fake implementation of DateTime Now, we can control the time e.g. for testing purposes

Composition of Narrow Components and Identifying and Extracting Responsibilities

![test](https://courses.dometrain.com/courses/take/from-zero-to-hero-writing-testable-code-in-csharp/lessons/63758906-composition-of-narrow-components)

To achieve narrow composition, you need to have classes that have a single responsibility.
There are usually two types of classes:
  - A model class
  - An "implementor" class e.g. service, manager class
- A model class can break single responsibiltiy by having so many methods inside it (God object)
- A service/manager class can also break single responsibility by having methods with so many if statements

To achieve narrow responsibility:
  - First just write bad code with big classes and if statements etc.
  - Then, critically evaluate each line and identify features that can be extracted out
    and extract them as their own class
  - For example, multiple if statements that govern how big a discount should be can be extracted
    to its own class called DiscountingRule or PromotionRule
  - Then, extract an interface out of this new class
  - Now the big class is smaller, so it adheres to SRP.
  - It is also easier to test the if statements now, since we don't need to instantitate a big class
    with all its dependencies just to test a single if statement! We just need to create dependencies
    relevant to the promotion rule relevant to that if statement in the new class.

The Identifying and Extracting Responsibility shows example of how to do this for a model class
The Composition of narrow components show the same concept, but for "implementation" class

What is opvallend is that, for "implementor"-type classses, usually the smaller extracted things do not have to be in a separate file, but 
can be in the same file. For models, best to extract them out to new files.
Too many files can also cause confusion! So no need sometimes to have many files.

Injectable vs Newable objects

https://courses.dometrain.com/courses/take/from-zero-to-hero-writing-testable-code-in-csharp/lessons/63758905-creating-effective-injection-points

- Not everything must be injected! Sometimes it's oke to have new() in your function
- How to know if dependency used inside a function can be instantiated vs. injected?
- If the dependency is not dependent on outside resources e.g. API call or db, then it's oke to instantiate, otherwise inject!
- e.g. in below, shipmentInfo is something not dependent on outside dependencies, so okay to new, but not the case for shippingRateCalculator ^mnk4fUw5
