## Text Elements
IAM Roles:

- A type of identity
- Difference to IAM User is that it can be 
assumed by multiple principles (e.g. multiple
persons, apps, AWS service like lambda, etc.)
- An IAM Role identity is assumed by another
identity e.g. a IAM User

- Just like a IAM user can be attached a policy,
a IAM Role can be attached policies too

- A IAM role identity is attached two types of policies:
trust policy (which principals are allowed to assume the role)
managed policies/inline policies (the actual things the assumer of the role can do)
 ^BZfI1tFW

When is a role needed?
- When a principal cannot be attached a policy directly
- For example: AWS lambda service, 
is a principal, that cannot have policies attached to it directly

- A Lambda service, to be useful, needs to have a lambdaexecutionpolicy 
attached
- So, we need to create a LambdaExecutionRole, with trust policy of
lambda.amazonaws.com, and managedpolicy of lambdaexecutionpolicy (see
beside for example)

- In the trust policy section, you need to explicitly say that this role
has access to all resources that the managedpolicy/inlinepolicy is allowed
to access
- This is done by specifying Action: sts:AssumeRole operation

- sts stands for secure token service
- This AWS service is responsible for generating temporary security creden
tials so that the role can access the resources it needs for its 
managedpolicy/inlinepolicy to work properly ^d4gupnLw

ignore these words; dont let them confuse you ^rWHe4EBf
